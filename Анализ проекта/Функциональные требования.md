## Управление тест-кейсами
### Структура тест-кейса (Поля)
**Кнопка "Сохранить"** - при нажатии сохраняет введённые данные и записывает изменения в историю версий.
**Поле "Секция"** - содержит название секции в которой располагается кейс. При нажатии отображает существующие секции, нажав на которые можно перенести кейс в иные секции.
**Поле с отображение текущей версии** - при нажатии можно изменить версию тест-кейса на более старую или новую.
**Поле "Теги"** - при вводе значений показывает теги соответствующие введённому значению, или предлагает создать новый. Тегов может быть неограниченное количество.
**Поле "Время прохождения"** - по умолчанию (5 минут),  пользователь вводит примерное время прохождения кейса.
- **Обязательные поля (без них кнопка "Сохранить" неактивна):**
	- Название (макс. 255 симв.)
	- Статус  (Готов/Требует доработки/ Не готов *(по умолчанию)* )
	- Приоритет ( Очень высокий/Высокий/Средний  *(по умолчанию)* /Низкий/Очень низкий)
	- Параметры (появляется при использовании параметра (%*Название_параметра*) )
- **Основной контент** (RIch Text Editor):
		Можно добавлять/удалять строки для шагов, пред- и пост- условий кейса.
		Можно дублировать строку для шагов, пред- и пост- условий кейса.
		Можно добавить строку Общего шага, для которого применяется отдельная нумерация ( строка общего шага содержит:  № (по общей нумерации строки), Название общего шага, ID шага в проекте с гиперссылкой на этот шаг)
	- Предусловие секции - наследуется из секции, в которой находится тест-кейс(1 уровень) - **не редактируется**.
	- Предусловие кейса - состоит из 3 колонок (№, Действие, Ожидаемый результат)
	- Шаги кейса - состоит из 3 колонок (№, Действие, Ожидаемый результат)
	- Постусловие кейса - состоит из 3 колонок (№, Действие, Ожидаемый результат)
	- Постусловие секции - наследуется из секции, в которой находится тест-кейс(1 уровень) - **не редактируется**.
- **Вложения:**
	- Возможность прикрепить скриншот/файл к отдельному шагу
	- Drag&Drop загрузка файлов. Ограничение 1 ГБ на файл
- **История Результатов:**
	- Содержит историю результатов прохождения тест-кейса в рамках тест-плана
- **История Изменений:**
	- Содержит историю изменения тест-кейса
- **Ссылки:**
	- содержит строки с полями:
		- Тип (Дефект, Документация, Связанный кейс)
		- Ссылка
- **Комментарии:**
	- содержит поле для ввода комментария и историю переписки
- **Связанные автотесты:**
	- Содержит поле в котором можно связать кейс с автотестом, а также список связанных автотестов
- **Мета-данные:**
	- Автор
	- Дата Создания
	- Дата последнего изменения
	- Версия тест-кейса

## Управление тестовыми прогонами

## Создание прогона
- **Источник кейсов:**
    
    - Выбор конкретной папки из библиотеки (со всей вложенностью).
        
    - Выбор по динамическому фильтру (например, «Все кейсы с приоритетом Высокий, изменённые за последнюю неделю»).
        
    - Ручное добавление отдельных кейсов через поиск.
        
- **Параметры запуска:**
    
    - Название прогона (обязательно, например: «Регресс релиза v2.3.0»).
        
    - Окружение (Environment): поле для ввода (Staging, Prod, iOS 17.4).
        
    - Сборка (Build): версия тестируемого билда.
        
    - **Критически важно:** В момент создания прогона система делает **слепок (snapshot)** всех выбранных тест-кейсов. Все дальнейшие изменения в библиотеке **не должны** влиять на этот прогон.
        
- **Автоматическое присвоение ID:** Прогон получает уникальный идентификатор (например, `TR-42`).
**Нюанс:** Возможность указать **время старта** в будущем. Иногда прогон создаётся в пятницу вечером, а запускается командой в понедельник утром.
## Интерфейс выполнения
- **Режимы отображения:**
    
    - **Пошаговый (Step-by-step):** На экране один шаг. Крупные кнопки **Успешно** / **Провалено** / **Блокировано**.
        
    - **Табличный (Grid):** Видны все шаги кейса сразу, статус проставляется в выпадающем списке.
        
- **Статусы выполнения (на уровне шага):**
    
    - `Не пройдено (по умолчанию).
        
    - `Успешно`.
        
    - `Провалено` — при выборе **обязательно** открывается форма создания дефекта (см. п. 4.3).
        
    - `Блокировано` — шаг нельзя выполнить из-за бага на предыдущем шаге или проблем окружения. Требуется комментарий.
        
    - `Пропущено` .
        
- **Фактический результат:** Для статуса `Провалено` появляется текстовое поле «Фактический результат» (обязательное).
    
- **Вложения:** Возможность прикрепить скриншот/видео к **конкретному шагу** (часто нужно показать ошибку).
    
- **Навигация:**
    
    - Кнопки «Следующий кейс» и «Предыдущий кейс».
        
    - Индикатор прогресса: «Кейс 5 из 23».
        
    - Возможность **пропустить** кейс и вернуться к нему позже (оставить в статусе `Не пройдено`).
        
- **Автосохранение:** Каждое изменение статуса шага должно сохраняться мгновенно (без нажатия кнопки «Сохранить»).
    

**Нюанс:** «Таймер прохождения». При начале прохождения тест-кейса, включается таймер (его можно поставить на паузу, воспроизвести, перезапустить).
## Назначение исполнителей
- **Модели назначения:**
    
    - **Весь прогон на одного:** Классический вариант. Один человек отвечает за всё.
        
    - **Распределение по кейсам (Assignment):** Возможность массово выделить кейсы и назначить на конкретного пользователя.
        
- **Мой список (My Tasks):** У каждого QA на главном экране должен быть виджет «Назначено мне», где видны **все** прогоны, в которых у него есть неназначенные или назначенные лично ему кейсы.
    
- **Уведомления:** При назначении кейса на пользователя ему приходит Push / Email (в зависимости от настроек профиля) с прямой ссылкой на этот кейс.
## Управление прогрессом
Жизненный цикл прогона: Ожидает -> В процессе -> Завершён -> Архивирован.
- **Статусы прогона (верхний уровень):**
    
    - `В процессе` (по умолчанию после создания).
        
    - `Приостановлен` (если тестирование приостановлено).
        
    - `Выполнен` (все кейсы имеют финальный статус Passed/Failed/Skipped).
        
    - `Архивирован` (архивный, результаты зафиксированы).
        
- **Перезапуск (Re-run):**
    
    - Упавший кейс можно пометить как «Требуется перепроверка». При этом **не создаётся новый прогон**, а старый кейс переводится обратно в `Не пройдено`.
        
    - В истории кейса сохраняется вся цепочка: первая попытка (Провал) -> вторая попытка (Пропущено).
        
- **Блокировка кейса:**
    
    - Если кейс в статусе `В процессе` у одного пользователя, другой пользователь **не может** начать его проходить (оптимистичная блокировка с предупреждением).
## Завершение прогона
- Завершение:** При переводе в `Выполнено` система проверяет, что все шаги всех кейсов имеют статус (нет `Не пройдено`). Если есть — либо блокируем закрытие, либо предлагаем массово проставить `Пропущено`.
    
- **Отчёт по прогону (Run Summary):**
    
    - Пройдено кейсов: X / Y.
        
    - Pass Rate: Z%.
        
    - Список всех созданных дефектов (с ID).
        
    - Время выполнения (если включён таймер).
        
- **Экспорт:** Возможность скачать отчёт в формате **PDF** или поделиться **ссылкой для гостей** (только чтение).
# Работа с дефектами
## Создание дефекта из тест-кейса
- **Контекстная передача данных:** При нажатии кнопки «Создать дефект» на шаге, в форму создания автоматически подставляются:
    
    - Название (Summary): `[Прогон TR-42] Название тест-кейса`.
        
    - Описание (Description): Ссылка на кейс, **все шаги + ожидаемые результаты** (форматированный текст).
        
    - Окружение: Сборка, ОС, браузер (взяты из параметров прогона).
        
    - Фактический результат: То, что ввёл пользователь в поле Fail.
        
- **Прикрепление скриншотов:** Если QA прикрепил изображение к шагу, оно должно либо сохраниться в хранилище TMS и отдаваться по ссылке, либо передаваться вложением во внешнюю систему (если API позволяет).
    
- **Связывание:** После успешного создания тикета (например, в Jira), в TMS в карточке прогона рядом с кейсом появляется кликабельный **ID дефекта** (`PROJ-123`).
    

**Нюанс для Jira:** В Jira есть поле `Environment`. Нужно убедиться, что в вашем проекте Jira это поле текстовое, и мы можем записать в него данные о сборке. Иначе они потеряются.
## Статусы
- **Отображение статуса:**
    
    - `To Do` / `Open` -> В TMS дефект помечается серым/синим.
        
    - `In Progress` -> Жёлтым.
        
    - `Resolved` / `Ready for QA` -> **Зелёным (Требует перепроверки).**
        
    - `Closed` / `Done` -> Зачёркнутый текст.
        
- **Действие «Перепроверить исправление»:**
    
    - Когда в Jira статус меняется на _«Готово к тестированию»_, в TMS у соответствующего кейса должна появиться кнопка **«Re-test»**.
        
    - При нажатии QA переходит в режим повторного прохождения именно этого кейса.
        

**Критический нюанс:** **Идемпотентность запросов.** Если пользователь дважды нажал «Создать дефект», в Jira **не должно** создаться два одинаковых тикета. TMS должна проверять, нет ли уже привязанного открытого бага к этому кейсу.
## Вкладка «Дефекты» в прогоне и проекте
- **Агрегация:** На странице Test Run есть вкладка **Defects**.
    
- **Содержание таблицы:** ID дефекта, Заголовок, Привязанный кейс, Статус в Jira, Дата обнаружения.
    
- **Фильтр «Блокирующие»:** Возможность отфильтровать только те баги, из-за которых кейсы находятся в статусе `Blocked`.

# Отчётность и дашборды
## Дашборд проекта
- **Виджет 1: Статистика по библиотеке кейсов (Coverage):**
    
    - Общее количество кейсов в проекте.
        
    - Количество **готовых** кейсов (`Готов`).
        
    - Количество **Не готовых** (`Не готов`).
        
    - Распределение по приоритетам (круговая диаграмма: Очень высокий / Высокий / Средний / Низкий / Очень низкий).
        
- **Виджет 2: Текущая активность (Active Runs):**
    
    - Таблица со списком **активных прогонов** (со статусом `В процессе`).
        
    - Для каждого прогона: Название, Дата старта, Прогресс (Pass Rate %), Количество оставшихся кейсов.
        
- **Виджет 3: Последние найденные дефекты:**
    
    - Список из 5-10 последних багов, созданных в рамках проекта.
        
    - Отображение ID дефекта, Статуса в Jira и ФИО нашедшего.
- **Виджет 4: Участники:**
	- Количество созданных тест-кейсов
	- Количество созданных автотестов
	- Количество успешных прохождений тест-кейсов
	- Количество заведённых дефектов
	- Количество редактированных кейсов
        
- **Фильтрация по датам:** Все виджеты должны поддерживать выбор временного периода (Последние 7 дней, 30 дней, Всё время).

** Нюанс:** Скорость загрузки дашборда. Данные для виджетов должны кэшироваться или считаться асинхронно, чтобы пользователь не ждал 5 секунд при каждом открытии страницы.
## Аналитика по прогону
- **Метрики верхнего уровня (KPI Cards):**
    
    - **Процент прохождения:** (Пройдено / Всего) * 100%.
        
    - **Количество дефектов:** Количество созданных дефектов.
        
    - **Количество блокированных:** Количество заблокированных тестов.
        
    - **Время прохождения:** Суммарное время, затраченное на прохождение (если включён трекинг времени).
        
- **Таблица результатов по кейсам:**
    
    - Колонки: ID кейса, Название, Исполнитель, Статус выполнения, Ссылки на дефекты.
        
    - Возможность **развернуть строку** и посмотреть историю шагов (Успешно/Провалено).
        
- **Диаграмма распределения статусов:**
    
    - Круговая диаграмма: Успешно, Провалено, Блокировано, Пропущено, Не пройдено.
## Метрики качества
- **График Pass Rate по времени:**
    
    - Ось X: Прогоны (по дате старта).
        
    - Ось Y: Процент Прохождения.
        
    - **Интерактивность:** При клике на точку графика открывается тот самый прогон.
        
- **График плотности дефектов:**
    
    - Количество багов, созданных за временной период (столбчатая диаграмма).
        
- **Тепловая карта покрытия требованиями (Roadmap Item):**
    
    - _Важно:_  в версии 1.0 этот функционал **не реализуется**, но в структуре БД должна быть предусмотрена связь «Тест-кейс -> ID Требования» для будущего расширения.
##  Экспорт данных
- **Экспорт в CSV/Excel:**
    
    - Выгрузка данных из таблицы результатов прогона.
        
    - Поддержка UTF-8 (чтобы не слетали русские символы).
        
- **Публичная ссылка (Shareable Link):**
    
    - Возможность сгенерировать уникальную ссылку на страницу с результатами прогона **только для чтения**.
        
    - Ссылка доступна без авторизации (или с ограничением по времени / количеству просмотров).
        
    - **Опция:** Скрывать или показывать в публичном отчёте названия кейсов (иногда это NDA-информация).
# Администрирование
## Проекты
- **Создание проекта:**
    
    - Поля: `Название` (обязательное, уникальное в рамках инстанса), `Код проекта` (латиница, для префикса ID кейсов, например `PROJ`), `Описание`.
        
    - При создании автоматически создаётся корневая папка для тест-кейсов.
        
- **Настройки проекта:**
    
    - Список доступных **Окружений** (Environment) — настраиваемый справочник (например: `Dev`, `Staging`, `Prod`).
        
    - Шаблон названия прогона по умолчанию (например, `[Build] - [Date] Regression`).
        
    - Возможность включить/отключить обязательность поля `Build` при старте прогона.
        
- **Архивирование:**
    
    - Администратор может перевести проект в статус `Архивирован`. Архивные проекты не видны в общем списке, но данные не удаляются.
        
    - Восстановление из архива возможно.
        

**Нюанс:** В коммерческих SaaS-решениях часто есть ограничение на количество активных проектов в зависимости от тарифа.
## Пользователи
- **Приглашение пользователя:**
    
    - Форма с полем `Email`.
        
    - Выбор **глобальной роли** (Админ, Тимлид, QA-инженер ) — см. Пункт 3.
        
    - Опционально: выбор конкретных проектов, к которым будет доступ. Если не выбрано — доступ ко всем проектам (кроме Admin, у него всегда ко всем).
        
- **Управление учётными записями:**
    
    - Таблица пользователей с фильтрацией по роли и статусу (Active / Pending / Blocked).
        
    - Возможность деактивировать пользователя (лишить доступа) без удаления его исторических данных.
        
    - Редактирование роли пользователя.
        
- **Аутентификация:**
    
    - Базовая: Email + пароль (с требованиями к сложности).
        
    - Восстановление пароля через ссылку на почту.
        
    - **Для будущих версий:** Интеграция с корпоративным SSO (SAML/OAuth), но в MVP не включается.
        

**Нюанс:** Действие «Удалить пользователя» должно быть недоступно или скрыто под дополнительным подтверждением, так как это может сломать связи «Автор кейса», «Исполнитель прогона».
## Импорт
- **Поддерживаемые форматы:**
    
    - **CSV / Excel (XLSX)** — основной формат.
        
    - **JSON** — для миграции из других TMS через API.
        
- **Процесс импорта из CSV/Excel:**
    
    1. Загрузка файла.
        
    2. **Mapping (Сопоставление полей):** Интерфейс, где пользователь указывает, какая колонка в файле соответствует какому полю в TMS (`Название`, `Шаги`, `Предусловие`, `Постусловие`).
        
    3. **Валидация:** Проверка на пустые обязательные поля, проверка формата шагов (если шаги в одной ячейке).
        
    4. **Предпросмотр:** Показать 5-10 записей, как они будут выглядеть после импорта.
        
    5. **Импорт:** Создание кейсов в выбранной папке проекта.
        
- **Обработка ошибок:**
    
    - При ошибке валидации импорт останавливается, пользователю выдаётся лог с номерами строк и причинами ошибок.
        
    - Успешно импортированные записи сохраняются.
        

** Нюанс:** **Шаги в Excel.** Самая частая боль — как парсить шаги. Нужно поддержать два сценария:

1. Шаги в одной ячейке с разделителем (например, `1. Открыть страницу -> Должна загрузиться`).
    
2. Каждая пара «Действие-Ожидаемый результат» в отдельных колонках (`Step Action`, `Step Expected`).
